# プロンプト管理仕様

## 1. 目的

LLM に渡す Summary 生成用、回答例生成用のプロンプトはコードに埋め込まず、ファイルとして外出しして管理する。これにより、Go コードを変更せずに文言、制約、出力トーンを調整できるようにする。

## 2. 管理方針

- プロンプトは `prompts/` ディレクトリで管理する
- タスクごとに `system` と `user` を分ける
- 拡張子は `.tmpl` を推奨する
- Go の `text/template` を第一候補とする
- 実行時に使用したプロンプトのファイル名とハッシュ値を保存する
- 本番運用ではプロンプト変更時にレビューを必須にする
- preview や一時的なレンダリング結果の保持期間は環境変数で制御し、既定値は 30 日とする

## 3. 推奨構成

```text
prompts/
  period_summary/
    system.tmpl
    user.tmpl
  account_report/
    system.tmpl
    user.tmpl
```

## 4. テンプレートに含めるべき要素

### 4.1 system.tmpl

- 役割定義
- 禁止事項
- 出力形式の制約
- 幻覚防止ルール
- 回答言語

### 4.2 user.tmpl

- 実行対象の期間やアカウント
- 課題一覧
- コメント要約
- 出力 JSON Schema
- 補足指示

## 5. 変数の例

| 変数名 | 用途 |
| --- | --- |
| `ProjectKey` | Backlog プロジェクト識別 |
| `ProjectName` | 表示名 |
| `DateFrom` | 開始日 |
| `DateTo` | 終了日 |
| `AccountID` | 対象アカウント識別 |
| `AccountName` | 対象アカウント表示名 |
| `IssuesJSON` | LLM に渡す課題一覧 JSON |
| `OutputSchemaJSON` | 期待する JSON Schema |
| `Language` | 出力言語 |

## 6. 実行時要件

- 起動時に `PROMPT_DIR` を解決する
- 必須ファイルが欠けていれば即時エラーにする
- テンプレートの構文エラーは起動時または初回ロード時に検出する
- `--prompt-dir` フラグでプロンプトディレクトリを上書きできるようにする
- `--dry-run` 時はレンダリング後のプロンプトを標準出力または `PROMPT_PREVIEW_DIR` に保存できるようにする
- `PROMPT_ARTIFACT_RETENTION_DAYS` の既定値は `30` とする

## 7. バージョン管理

- プロンプトは Git 管理対象に含める
- 変更時は差分レビューを必須にする
- ジョブ実行履歴には次を保存する
  - プロンプト種別
  - 使用ファイル名
  - SHA-256 などのハッシュ
  - 実行日時

## 8. テスト方針

- テンプレートのレンダリングテスト
- 最低限の入力で期待する JSON Schema 指示が含まれることの確認
- 危険な表現や禁止文言が混入していないことの確認
- スナップショット比較による変更影響の確認

## 9. 運用上の注意

- 課題本文やコメントの生データをそのまま長文で埋め込まない
- 個人情報、秘密情報、アクセストークンらしき値は Slack 出力前にマスクする
- プロンプト調整時は本番前に `--dry-run` とステージング Slack で確認する
- `--dry-run` では system / user の両方の最終レンダリング結果を確認できるようにする
- モデルごとの差分が大きい場合は `openai/` と `gemini/` のサブディレクトリ分割も検討する
- LLM に渡す本文やコメントには個人情報が含まれていてもよいが、Slack 出力には同じ内容をそのまま使わない
