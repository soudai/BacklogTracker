# 要件定義

## 1. 背景

Backlog 上の課題とコメントを人手で追い、期間別の状況や担当者ごとの対応状況を整理する作業には時間がかかる。本システムは Backlog API から必要データを収集し、LLM で要約・回答例作成を行い、その結果を Slack に通知することで、状況把握と初動対応を高速化する。

## 2. スコープ

### 2.1 対象機能

- 指定期間の課題サマリ生成
- 指定アカウントに関連する課題一覧の取得
- 対象課題ごとの要点整理
- 課題コメントに対する回答例の提案
- 実行時の LLM プロバイダ切り替え
- Slack 通知
- Ubuntu 上の CLI コマンド実行
- SQLite とテキストファイルへの結果保存

### 2.2 対象外

- Backlog 課題やコメントの自動更新
- Slack 以外への通知
- 複数 Backlog スペースの横断集計
- LLM 生成結果の自動承認ワークフロー

## 3. 利用者

| 利用者 | 目的 |
| --- | --- |
| プロジェクト管理者 | 指定期間の課題の傾向、未完了事項、注意点を確認する |
| 担当者 / チームリーダー | 指定アカウントの課題状況と返信のたたき台を確認する |
| 開発者 / 運用者 | Ubuntu 環境で CLI を実行し、API キー、モデル、通知先、保存先を設定して安定運用する |

## 4. ユースケース

### UC-01: 指定期間の課題サマリを受け取る

- 入力
  - Ubuntu 上での CLI サブコマンド実行
  - プロジェクト ID またはプロジェクトキー
  - 集計期間
  - 期間の基準種別（`created` または `updated`。既定値は `updated`）
  - 任意のフィルタ（ステータス、優先度、担当者など）
  - 使用 LLM プロバイダ
- 出力
  - 対象件数
  - ステータス別件数
  - 優先度の高いトピック
  - 要注意チケット
  - Slack 通知メッセージ
  - SQLite への実行履歴保存
  - テキストファイルへのレポート保存

### UC-02: 指定アカウントの課題と回答例を受け取る

- 入力
  - Ubuntu 上での CLI サブコマンド実行
  - プロジェクト ID またはプロジェクトキー
  - アカウント ID または userId
  - 取得期間
  - 対象条件（assignee 課題のみ）
  - 使用 LLM プロバイダ
- 出力
  - 対象アカウントの課題一覧
  - 各課題の要約
  - 必要なら直近コメントの要点
  - 回答例の提案
  - Slack 通知メッセージ
  - SQLite への実行履歴保存
  - テキストファイルへのレポート保存

### UC-03: Slack に通知する

- 入力
  - 通知先チャンネル
  - メッセージ形式
  - 失敗時の再送方針
- 出力
  - 通知成功 / 失敗結果
  - 送信したジョブ ID

### UC-04: Ubuntu 上でコマンドとして実行する

- 入力
  - サブコマンド
  - 各種フラグ
  - 環境変数
- 出力
  - 標準出力または標準エラーへの実行結果
  - 終了コード
  - 実行履歴の永続化

### UC-05: 初期セットアップを行う

- 入力
  - `init` サブコマンド
  - 対話入力または既存環境変数
- 出力
  - `.env.local`
  - SQLite 初期化結果
  - 利用可能なサンプルプロンプト

## 5. 機能要件

### 5.1 Backlog データ取得

- FR-01: プロジェクトを指定して課題一覧を取得できること
- FR-02: 指定期間で課題を絞り込めること
- FR-03: ページネーションを処理し、全件取得できること
- FR-04: 指定アカウントで課題を絞り込めること
- FR-04a: 指定アカウントの関連課題は初期実装では assignee のみを対象にすること
- FR-05: 課題詳細に加えてコメント一覧を取得できること
- FR-06: ステータスや担当者などの名称解決に必要な補助データを取得できること

### 5.2 要約生成

- FR-07: 指定期間の課題群から全体要約を生成できること
- FR-08: 課題ごとの要約を生成できること
- FR-09: 出力は日本語を既定とし、将来の多言語化に備えて言語指定可能な構造にすること
- FR-10: LLM の出力は JSON スキーマで検証できること
- FR-11: 課題本文やコメントに存在しない事実を確定情報として出力しないこと

### 5.3 回答例提案

- FR-12: 直近コメントや課題の文脈を基に回答例を 1 件以上提案できること
- FR-13: 回答例には前提不足や確認事項がある場合、その旨を含められること
- FR-14: 回答例は Backlog へ自動投稿しないこと

### 5.4 プロバイダ切り替え

- FR-15: 実行時に `gemini` または `chatgpt` を指定できること
- FR-16: プロバイダ差分は共通インターフェースで吸収し、呼び出し側は同一フォーマットで扱えること
- FR-17: プロバイダごとのモデル名、タイムアウト、リトライ回数を設定できること

### 5.5 Slack 通知

- FR-18: 要約結果を Slack に送信できること
- FR-19: 通知は最小限でも本文テキストを持ち、必要に応じて Block Kit 形式を使えること
- FR-19a: Slack 通知は Block Kit を基本とし、フォールバック用 `text` を必ず持つこと
- FR-20: 通知失敗時はリトライし、最終失敗をログに残せること

### 5.6 CLI 実行

- FR-21: Ubuntu 上で単一の CLI バイナリとして実行できること
- FR-22: `init`、`period-summary`、`account-report` のサブコマンドを提供すること
- FR-23: 引数と環境変数の両方で設定を受け取れること
- FR-24: 正常終了と異常終了を終了コードで判定できること
- FR-25: `init` は対話的に初期設定を入力できること
- FR-25a: `init` では `gemini` または `chatgpt` のどちらかを必須選択し、既定値は `gemini` とすること
- FR-26: `init` は `.env.local` と必要ディレクトリを生成できること
- FR-27: `init` は SQLite マイグレーションを実行できること

### 5.7 永続化

- FR-28: 実行履歴、通知結果、ジョブ状態を SQLite に保存できること
- FR-29: 生成した要約結果と回答例をテキストファイルとして保存できること
- FR-30: 必要に応じて raw response やエラーレポートをテキストファイルに保存できること

### 5.8 プロンプト管理と設定

- FR-31: Summary 生成用と回答例生成用のプロンプトをファイルとして外出しできること
- FR-32: プロンプトファイルはコード変更なしで調整できること
- FR-33: API Key や各種接続設定を `.env`、`.env.local`、または環境変数から取得できること
- FR-34: 実行時に利用したプロンプトの識別情報を記録できること
- FR-35: `--dry-run` 時にレンダリング後のプロンプトを確認できること
- FR-36: prompt / preview の保持期間を環境変数で指定でき、既定値は 30 日であること

## 6. 非機能要件

- NFR-01: API キーやトークンは環境変数またはシークレット管理基盤から読み込む
- NFR-02: 機密情報をログに出力しない
- NFR-03: Backlog / LLM / Slack の失敗を切り分けられる構造化ログを出力する
- NFR-04: Backlog API のレート制限を考慮して呼び出し回数を制御する
- NFR-05: LLM 応答の揺れに備えてスキーマ検証とフォールバックを実装する
- NFR-06: 同一ジョブの重複通知を避けるため、ジョブ ID ベースの冪等性を持たせる
- NFR-07: タイムゾーンは設定可能とし、既定値は `Asia/Tokyo` とする
- NFR-08: 実行履歴と通知結果を後から追跡できること
- NFR-09: Ubuntu 上で追加の常駐プロセスなしに CLI 実行できること
- NFR-10: Go 製の単一バイナリ配布を基本とすること
- NFR-11: SQLite ファイルとテキスト出力先は設定可能であること
- NFR-12: Cron や systemd timer など外部スケジューラから起動しやすいこと
- NFR-13: `.env` / `.env.local` 未設定や必須環境変数不足は起動直後に検知できること
- NFR-14: プロンプトファイルの構文エラーや不足を起動時または初回読込時に検知できること
- NFR-15: `init` は未設定項目を案内し、初回利用者が手順書なしでも初期化できること
- NFR-16: `.env.local` は Git 管理対象に含めず、テンプレートから生成すること
- NFR-17: Slack には Backlog 本文の生データを送らないこと
- NFR-18: Slack に送る個人情報はマスキングまたは除去すること
- NFR-19: `dry-run` の副作用なし動作と終了コードはテストで担保すること

## 7. 受け入れ条件

### AC-01 指定期間サマリ

- 期間とプロジェクトを指定してジョブ実行できる
- 取得した課題件数が Slack メッセージに含まれる
- 要約本文に重要トピックと要注意項目が含まれる
- LLM が失敗してもジョブ失敗理由が確認できる

### AC-02 指定アカウントサマリ

- 対象アカウントを指定して関連課題を抽出できる
- 各課題に対して短い要約が作られる
- 回答例がスキーマに沿って返る
- 回答例が未確定情報を断定していない

### AC-03 プロバイダ切り替え

- 同一入力で `gemini` と `chatgpt` を切り替えて実行できる
- 呼び出し側の保存形式が変わらない
- 失敗時にどのプロバイダで失敗したか識別できる

### AC-04 Slack 通知

- 正常時に指定チャンネルへ通知できる
- 送信ペイロードに Block Kit と最低限のプレーンテキストが含まれる
- Backlog 本文の生データが Slack に送られていない
- 失敗時に再送回数と最終結果が記録される

### AC-05 CLI 実行と保存

- Ubuntu シェルからサブコマンドを指定して実行できる
- 実行結果に応じて適切な終了コードが返る
- ジョブ履歴が SQLite に保存される
- 要約または回答例がテキストファイルとして保存される

### AC-06 init コマンド

- `init` 実行時に必要項目を対話的に入力できる
- LLM プロバイダは `gemini` または `chatgpt` から必須選択でき、既定値は `gemini`
- `.env.local` がテンプレートベースで生成される
- SQLite の初期マイグレーションが適用される
- 初期プロンプトと出力ディレクトリが利用可能な状態になる
